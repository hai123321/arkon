1 Executive Summary

The bank operates dozens of large systems (core banking, payments, lending, channels…). Design documentation is scattered across Confluence, Word, email, and the heads of senior architects. Every production deployment widens the gap between code and documentation.

Proposal: Build a Master Design Vault — a centralized, version-controlled documentation repository on GitLab, automatically updated by AI after each production deployment, with developers interacting through Kiro IDE as the organization's "second brain".

2 Problem Statement

Current pain points

Banking-specific risks

  • Compliance: Audits require proving "code A matches design B" — currently a manual scramble
  • Senior departure risk: Knowledge of core banking integration walks out the door
  • Incident response: Locating architecture specs during a prod incident takes 30+ minutes
  • Decision drift: 6 months later, no one remembers why pattern X was chosen
  • Cross-system duplication: Teams A and B solve the same problem unaware of each other

3 Proposed Solution — High Level Design

The big picture — 4-layer architecture

📥
Layer 1 — Data Sources
Already exist in the bank
GitLab repos · Jira tickets · Confluence specs · API contracts · DB schemas · ADRs · Postmortems · CI/CD pipelines · Runbooks
↓ Webhooks + scheduled scan
🧠
Layer 2 — AI Ingestion Pipeline
On-prem, self-hosted
  • Trigger: post-deploy hook from Change Management Process (via GitLab CI)
  • Diff extractor: only ingest changes (no full re-scan)
  • Redactor: strip PII, PCI, account numbers (regex + NER)
  • AI Engine: Claude on AWS Bedrock — all data stays inside the bank's VPC, no internet egress
  • Audit log every AI call via AWS CloudTrail (compliance-ready)
↓ Create Merge Request (NO auto-merge)
📚
Layer 3 — Master Design Vault
GitLab repo, markdown-first
master-design/
├── _index.md                ← AI-maintained TOC
├── systems/
│   ├── core-banking/        ← 1 folder per system
│   │   ├── overview.md
│   │   ├── api-contracts.md
│   │   ├── data-model.md
│   │   ├── integrations.md
│   │   └── decisions/       ← auto-generated ADRs
│   ├── payment-gateway/
│   └── lending-system/
├── cross-cutting/           ← auth, logging, security
├── conflicts/               ← AI-flagged, awaiting review
└── .ai-instructions.md      ← AI "constitution"
↓ git pull / MR review
💻
Layer 4 — Interaction (Kiro IDE)
Developer interface
  • Open vault as a workspace in Kiro
  • Ask: "How does Payment call Core Banking?" → Kiro reads the vault, answers with source links
  • Steering files = .ai-instructions.md
  • Spec workflow: write a new spec → Kiro auto cross-references existing systems and warns of conflicts

The auto-update lifecycle

Core design principles

PrincipleRationale
Markdown + Git instead of ConfluenceCompliance, native audit trail, Kiro works best with file systems
AI never auto-mergesBanking requires human-in-the-loop. AI only proposes MRs; Brain Owner approves
Source links are mandatoryEvery entry must link back to original commit/Jira/Confluence → no hallucinations
Redaction before AIProtects PII, PCI, customer data
AI runs in-VPC via AWS BedrockNo data egress, CloudTrail audit logs, IAM-based access — regulator-friendly
Conflict detection, not auto-fixAI flags conflicts → humans decide

4 Technology Stack

LayerTechnologyWhy
Vault storageGitLab (self-hosted)Already in place, audit trail, MR review
AI engineClaude (Sonnet/Haiku) on AWS BedrockData stays in the bank's VPC, no egress — compliance-ready for banking
PipelineAWS Lambda / ECS + GitLab CISame AWS infrastructure as the vault, IAM-based auth
Audit logAWS CloudTrail + CloudWatchEvery Bedrock invocation is fully logged
InteractionKiro IDEFile-system native, AI-first
Vector search (post-MVP)Amazon OpenSearch / pgvector on RDSSemantic search over the vault, AWS-native
✅ Why AWS Bedrock wins: The bank already runs on AWS, so no separate egress proxy / DLP layer is needed — data stays in-VPC by default. Bedrock is an AWS-managed service with SOC 2, ISO 27001, PCI DSS, and HIPAA-eligible attestations. Anthropic does not store or train on the bank's data (per Bedrock Data Privacy Policy). Compliance is no longer a blocker.

8 Decision Requested

Approval requested for

  • Phase 0 + Phase 1 — 6-week pilot Roll out a pilot on one non-core-banking system to validate value and build the foundation for wider rollout.
  • MVP budget — only ~$9K total Allocate 1 senior × 3 months (~$7,500 internal, no fresh cash needed) + ~$1,500 actual new budget for AI API + buffer.
  • Enable AWS Bedrock (Claude models) Security/Cloud team enables Bedrock model access in the bank's AWS account — takes only 1-2 days, no Legal review required since data stays in-VPC.
  • Pilot selection — pick one specific system Recommend a system with the worst documentation but low blast radius (not core banking, not handling realtime transactions).

Subsequent decisions (post-Phase 1)

  • Expand to the next 5 systems
  • Pick model tier: Claude Sonnet (quality) vs Haiku (5× cheaper) per task type
  • Integrate Jira + Confluence for richer context
